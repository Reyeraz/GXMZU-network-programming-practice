实验报告：登录验证原理与查询功能工作过程


（1）登录验证原理说明

本项目采用 JWT（JSON Web Token）+ 拦截器 的无状态认证方案：

1. 登录获取Token：用户在前端输入用户名密码，POST 请求发送到 /user/login，后端 UserServiceImpl.login() 校验用户名存在性、用户状态、密码正确性（BCrypt比对），验证通过后调用 JwtUtil.generateToken() 生成 JWT Token（包含 userId、username、7天有效期），返回给前端。

2. 前端保存Token：前端收到 Token 后，通过 Pinia 的 useUserStore().setAuth() 将 Token、用户名、用户ID 存入 localStorage 持久化保存，同时更新响应式状态。

3. 请求自动携带Token：Axios 请求拦截器（request.js）在每次请求前从 localStorage 读取 Token，判断该接口不在白名单中时，自动添加 Authorization: Bearer <token> 请求头。

4. 后端拦截验证：JwtInterceptor.preHandle() 拦截所有非公开请求，从请求头提取 Token，调用 JwtUtil.validateToken() 验证签名和有效期，验证通过后解析出 userId 存入 request.setAttribute("userId", userId)，供 Controller 使用。

5. 放行规则：WebConfig 配置拦截器放行 /user/login、/user/register、/products/**、/categories/** 等公开接口，其余接口均需认证。


（2）查询功能的前后端工作过程说明

以商品分页查询为例：

1. 用户触发查询：用户在 ProductList.vue 页面选择分类、输入搜索关键词或翻页时，触发 fetchProducts() 方法，将当前页码、每页条数、分类、关键词作为参数调用 productAPI.getProductsByPage()。

2. 前端发送请求：api.ts 中的 API 方法通过 Axios 发送 GET 请求到 /api/products/page?page=1&pageSize=4&category=xxx&keyword=xxx，Vite 代理将 /api 前缀去掉后转发到后端 http://localhost:8080/products/page。

3. 后端接收处理：ProductController.getProductsByPage() 接收请求参数，调用 ProductService.getProductsByPage()，Service 层使用 MyBatis-Plus 的 LambdaQueryWrapper 构建查询条件（分类精确匹配、关键词模糊匹配名称/描述/品牌），配合 Page 分页对象执行 selectPage() 查询。

4. 数据库执行查询：MyBatis-Plus 分页插件自动生成 SQL，执行 SELECT * FROM product WHERE category = ? AND (name LIKE ? OR description LIKE ? OR brand LIKE ?) ORDER BY id DESC LIMIT ? OFFSET ?，返回商品列表和总记录数。

5. 响应返回前端：查询结果封装为 PageResult<Product>（含 items、total、page、pageSize），再通过 ApiResponse.ok() 包装为统一响应格式返回前端。

6. 前端渲染数据：Axios 响应拦截器提取 response.data，ProductList.vue 将 items 赋值给 products 响应式变量，total 赋值给分页组件，Vue 响应式系统自动更新商品列表和分页控件的显示。
