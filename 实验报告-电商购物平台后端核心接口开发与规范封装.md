# 实验报告：电商购物平台后端核心接口开发与规范封装

## 一、实验分析与设计

### 1.1 需求分析

本次实验要求完成电商购物平台后端的四项核心任务：

1. **统一返回结果类**：封装统一的 API 响应格式，包含状态码、消息、数据及分页信息
2. **RESTful 商品接口**：实现商品的增删改查基础接口
3. **全局异常处理**：统一处理参数校验错误、业务异常、资源不存在等异常情况
4. **接口测试**：编写测试用例覆盖正常流程与异常场景

### 1.2 技术方案设计

**技术栈选型**：
- Spring Boot 3.4.4 + Java 21
- MyBatis-Plus 作为 ORM 框架，简化数据库操作
- Lombok 减少样板代码
- Jakarta Validation 实现参数校验
- JUnit 5 + Mockito 进行单元测试

**架构设计**：采用标准三层架构（Controller → Service → Mapper），各层职责清晰：

```

│                   Client (前端)                   │
└───────────────────────┬─────────────────────────┘
```

### 1.3 统一返回结果类设计

设计 `ApiResponse<T>` 泛型类，包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | boolean | 请求是否成功 |
| `message` | String | 提示消息（失败时返回错误原因） |
| `data` | T | 响应数据（泛型，支持任意类型） |

提供两个静态工厂方法：
- `ApiResponse.ok(data)` — 成功响应，`success=true`，`message=null`
- `ApiResponse.fail(message)` — 失败响应，`success=false`，`data=null`

分页场景额外设计 `PageResult<T>` 类：

| 字段 | 类型 | 说明 |
|------|------|------|
| `items` | List\<T\> | 当前页数据列表 |
| `total` | long | 总记录数 |
| `page` | int | 当前页码 |
| `pageSize` | int | 每页大小 |

**设计考量**：采用泛型保证类型安全，`PageResult` 与 `ApiResponse` 解耦——分页信息作为 `data` 字段嵌套使用，便于扩展。

### 1.4 RESTful 商品接口设计

遵循 RESTful 规范，商品接口设计如下：

| HTTP 方法 | 路径 | 功能 | 请求体/参数 |
|-----------|------|------|-------------|
| GET | `/api/products` | 获取所有商品 | 无 |
| GET | `/api/products/{id}` | 根据 ID 查询商品 | 路径参数 id |
| GET | `/api/products/search?keyword=xx` | 搜索商品 | 查询参数 keyword |
| GET | `/api/products/page?page=1&pageSize=4` | 分页查询 | page, pageSize, category, keyword |
| GET | `/api/categories` | 获取所有分类 | 无 |
| POST | `/api/products` | 新增商品（JSON） | Product JSON |
| POST | `/api/products/add` | 新增商品（含图片） | ProductAddDTO + MultipartFile |
| PUT | `/api/products/{id}` | 更新商品 | Product JSON |
| DELETE | `/api/products/{id}` | 删除商品 | 路径参数 id |

**关键设计点**：
- 商品查询类接口（GET）在 `WebConfig` 中放行，无需登录即可访问
- 增删改操作需 JWT 认证
- 新增商品使用 `@Valid` + DTO 校验必填字段
- 分页查询使用 MyBatis-Plus 分页插件实现物理分页

### 1.5 全局异常处理器设计

通过 `@RestControllerAdvice` 注解实现全局异常捕获，分层处理不同类型的异常：

| 异常类型 | HTTP 状态码 | 场景 |
|----------|------------|------|
| `BadRequestException` | 400 | 参数错误、业务校验失败 |
| `ForbiddenException` | 403 | 权限不足 |
| `ResourceNotFoundException` | 404 | 资源不存在 |
| `MethodArgumentNotValidException` | 400 | Jakarta Validation 校验失败 |
| `RuntimeException` | 500 | 兜底处理，未预期的运行时异常 |

**设计原则**：
- 异常从具体到通用排列，避免被通用异常处理器提前捕获
- 所有异常统一转换为 `ApiResponse.fail()` 格式返回，保持前端解析一致性
- `MethodArgumentNotValidException` 提取第一条字段错误消息，避免返回冗长的校验细节

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class ApiResponse<T> {
    private boolean success;
    private String message;
    private T data;

    public static <T> ApiResponse<T> ok(T data) {
        return new ApiResponse<>(true, null, data);
    }

    public static <T> ApiResponse<T> fail(String message) {
        return new ApiResponse<>(false, message, null);
    }
}
```

### 2.2 商品 RESTful 接口实现

**Controller 层核心代码**：

```java
@RestController
@RequestMapping("/api")
public class ProductController {

    private final ProductService productService;

    // 获取所有商品
    @GetMapping("/products")
    public ApiResponse<List<Product>> getAllProducts() {
        return ApiResponse.ok(productService.getAllProducts());
    }

    // 根据ID获取商品
    @GetMapping("/products/{id}")
    public ApiResponse<Product> getProductById(@PathVariable Long id) {
        Product product = productService.getProductById(id);
        if (product != null) {
            return ApiResponse.ok(product);
        }
        return ApiResponse.fail("商品不存在");
    }

    // 分页查询商品
    @GetMapping("/products/page")
    public ApiResponse<PageResult<Product>> getProductsByPage(
            @RequestParam(defaultValue = "1") int page,
            @RequestParam(defaultValue = "4") int pageSize,
            @RequestParam(required = false) String category,
            @RequestParam(required = false) String keyword) {
        return ApiResponse.ok(productService.getProductsByPage(page, pageSize, category, keyword));
    }

    // 新增商品（含图片上传）
    @PostMapping("/products/add")
    public ApiResponse<Void> addProduct(@Valid @ModelAttribute ProductAddDTO dto,
                                        @RequestParam(value = "image", required = false) MultipartFile image) {
        productService.addProductWithImage(dto, image);
        return ApiResponse.ok(null);
    }

    // 更新商品
    @PutMapping("/products/{id}")
    public ApiResponse<Product> updateProduct(@PathVariable Long id, @RequestBody Product updated) {
        Product result = productService.updateProduct(id, updated);
        if (result != null) {
            return ApiResponse.ok(result);
        }
        return ApiResponse.fail("商品不存在");
    }

    // 删除商品
    @DeleteMapping("/products/{id}")
    public ApiResponse<Void> deleteProduct(@PathVariable Long id) {
        if (productService.deleteProduct(id)) {
            return ApiResponse.ok(null);
        }
        return ApiResponse.fail("商品不存在");
    }
}
```

**Service 层使用 MyBatis-Plus 分页插件**：

```java
public PageResult<Product> getProductsByPage(int page, int pageSize, String category, String keyword) {
    Page<Product> pageParam = new Page<>(page, pageSize);
    LambdaQueryWrapper<Product> wrapper = new LambdaQueryWrapper<>();
    if (category != null && !category.isEmpty()) {
        wrapper.eq(Product::getCategory, category);
    }
    if (keyword != null && !keyword.isEmpty()) {
        wrapper.and(w -> w.like(Product::getName, keyword)
            .or().like(Product::getDescription, keyword)
            .or().like(Product::getBrand, keyword));
    }
    wrapper.orderByDesc(Product::getId);
    Page<Product> pageResult = baseMapper.selectPage(pageParam, wrapper);
    return new PageResult<>(pageResult.getRecords(), pageResult.getTotal(), page, pageSize);
}
```

### 2.3 全局异常处理器实现

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 处理参数错误（400）
    @ExceptionHandler(BadRequestException.class)
    public ResponseEntity<ApiResponse<?>> handleBadRequest(BadRequestException e) {
        return ResponseEntity.status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.fail(e.getMessage()));
    }

    // 处理权限不足（403）
    @ExceptionHandler(ForbiddenException.class)
    public ResponseEntity<ApiResponse<?>> handleForbidden(ForbiddenException e) {
        return ResponseEntity.status(HttpStatus.FORBIDDEN)
                .body(ApiResponse.fail(e.getMessage()));
    }

    // 处理资源不存在（404）
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ApiResponse<?>> handleNotFound(ResourceNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(ApiResponse.fail(e.getMessage()));
    }

    // 处理 Jakarta Validation 校验失败（400）
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiResponse<?>> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
            .map(err -> err.getDefaultMessage())
            .findFirst()
            .orElse("参数校验失败");
        return ResponseEntity.status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.fail(message));
    }

    // 兜底处理未预期的运行时异常（500）
    @ExceptionHandler(RuntimeException.class)
    public ResponseEntity<ApiResponse<?>> handleRuntimeException(RuntimeException e) {
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(ApiResponse.fail(e.getMessage()));
    }
}
```

```java
        super(message);
    }
```

### 2.4 接口测试

#### 测试策略

采用 `@WebMvcTest` 进行 REST 层测试，Mock Service 层，专注于 Controller 逻辑和异常处理的验证。

**测试配置要点**：
- 使用 `@MockitoBean` Mock 掉 `ProductService` 和 `JwtInterceptor`
- 在 `@BeforeEach` 中模拟 JWT 拦截器放行，避免认证干扰测试
- `@ContextConfiguration` 中必须声明所有需要的 Bean（Controller、GlobalExceptionHandler、SecurityConfig、WebConfig）

## 三、实验总结

### 3.1 实验收获

1. **统一返回格式的重要性**：通过 `ApiResponse<T>` 泛型类，所有接口返回格式一致，前端只需一种解析逻辑，大幅降低了前后端联调成本。`PageResult` 与 `ApiResponse` 解耦的设计使分页信息可灵活扩展。

2. **RESTful 规范实践**：理解了 HTTP 语义化方法（GET 查询、POST 新增、PUT 更新、DELETE 删除）与资源路径设计原则。商品查询类接口放行认证，增删改需 JWT 验证，体现了安全与便利的平衡。

3. **全局异常处理机制**：`@RestControllerAdvice` 能捕获所有 Controller 层抛出的异常，避免在每个方法中重复 try-catch。异常处理器按从具体到通用的顺序排列，确保优先匹配精确异常类型。

4. **分层测试策略**：`@WebMvcTest` 只加载 Web 层，配合 Mockito Mock Service 层，测试启动速度快、隔离性好。通过 `@MockitoBean` Mock JWT 拦截器，解决了认证拦截器在测试环境中的干扰问题。

### 3.2 遇到的问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `@WebMvcTest` 测试报 Bean 注入失败 | `@ContextConfiguration` 未包含所有依赖的 Bean | 将 `GlobalExceptionHandler`、`SecurityConfig`、`WebConfig` 等全部加入 `classes` 属性 |
| JWT 拦截器阻断测试请求 | 测试环境无有效 Token | 在 `@BeforeEach` 中 Mock `JwtInterceptor.preHandle()` 返回 `true` |
| 分页查询返回全量数据 | 未配置 MyBatis-Plus 分页插件 | 注册 `MybatisPlusInterceptor` + `PaginationInnerInterceptor` Bean |
| `MethodArgumentNotValidException` 未被捕获 | 使用了 `@ModelAttribute` 而非 `@RequestBody`，触发的异常类型不同 | 全局异常处理器中增加对 `MethodArgumentNotValidException` 的捕获 |

### 3.3 注意事项

1. **异常处理器的顺序**：`@ExceptionHandler` 的匹配顺序很重要，应将最具体的异常类型放在前面，通用的 `RuntimeException` 放在最后作为兜底。
2. **DTO 与 Entity 分离**：新增商品使用 `ProductAddDTO` 而非直接使用 `Product` 实体，避免不必要的字段暴露和潜在的安全风险。
3. **测试中的 Mock 策略**：REST 层测试应 Mock 掉所有外部依赖（Service、Interceptor），确保测试只关注 Controller 本身的逻辑。
4. **接口放行配置**：新增公开接口时必须在 `WebConfig.addInterceptors()` 的 `excludePathPatterns` 中注册，否则会被 JWT 拦截器拦截。
